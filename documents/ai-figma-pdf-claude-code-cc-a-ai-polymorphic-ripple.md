# AI + 人协作工作流: Figma + PRD → iOS 代码

## Context

**起点**: 你想把 "Figma 链接 + PRD PDF" 作为输入,让 Claude Code 直接产出可用的 iOS 代码,以提升协作效率。

**澄清的认知**: 你之前在纠结 "人定步骤 vs AI 定 plan"。这是一个错误的二分。正确模型是 **分工**:

| 维度 | 谁负责 |
|---|---|
| WHAT (做什么) | **人**: PRD + Figma + 约束 |
| HOW (怎么做) | **AI**: 规划 + 实现 |
| 把关 | **人**: 在 plan 和测试用例两个节点 review |

**本工作流的定位** (基于本轮提问的回答):
- 通用 iOS 工作流 (项目约定靠 CLAUDE.md 注入,工作流本身不绑死项目)
- feature 粒度 (多页面 + 联调)
- 强制 plan-first,人工 review plan + 测试用例
- 真机/模拟器验收: 人工
- AI 信息不足时提问,不猜

---

## 核心理念

### "探索 → 规划 → 审阅 → 执行 → 验证" 循环

```
[人] 准备输入 → [AI] 探索+规划 → [人] 审 plan → [AI] 实现+测试 → [人] 审测试用例 → [人] 真机验收
                            ↑________________ 卡住时提问 _________________↓
```

### 五条不可妥协的原则
1. **AI 卡住提问,不猜**: PRD 歧义 / Figma 缺失 / 技术选型不明 → 停下来问,不要带着假设硬干
2. **Plan 必审,不当橡皮图章**: 代码改贵,plan 改便宜。审 plan 时质疑架构、复用、拆分粒度
3. **多会话隔离 context**: 一个 feature 不要塞进单一会话,plan 和每个页面/模块各起一个会话,避免 context 压缩丢信息
4. **AI 出报告,人做决策**: 探索阶段 AI 在代码库考古产出**文字报告**(文件路径、调用链、数据流向),你看报告做技术选型;不是让 AI 直接拍板,也不是你自己 dive into 代码
5. **AI 按笼统指令完美执行,所以指令必须细到状态机**: AI 不懂业务,不会替你判断业务正确性。"接口失败 = UI 失败" 这种笼统话 AI 会按字面执行,但状态机里 `网络偶发失败` / `业务真失败` / `轮询超时` 是不同状态。**plan 阶段必须把状态机画出来**,不能让 AI 推断

### vibe coding 适用边界

| 场景 | 能不能 vibe |
|---|---|
| 纯 UI、独立组件、文案改动 | ✅ 可以 (但 Figma token 必须严格,见 Stage 1) |
| 涉及数据流 / 状态机 / 缓存 / 跨模块 | ❌ 必须搞清楚数据流,否则 bug 修不动 |

## 前置准备 (每个 iOS 项目一次性配置)

### 1. 项目根 CLAUDE.md (最关键的杠杆点)

每个 iOS 项目根目录维护一份 `CLAUDE.md`,Claude Code 启动时自动加载。必须覆盖:
- **架构概览**: 主要模块 / 目录结构
- **关键依赖**: 网络层 / 路由 / UI / 数据库 / 分析 (附带"必须用什么、禁止用什么")
- **命名约定**: 文件命名 / API enum 命名 / 路由命名
- **硬性约束**: 例如 "网络一律走 BUDNetworking,不能直接 URLSession"、"导航必须走 Router,不能 push"
- **测试约定**: 用什么框架、写哪些类型的测试

参考样本: [Voca 的 CLAUDE.md](/Users/max/WorkSpace/VocaAI_iOS/CLAUDE.md) — 已经覆盖以上所有项,可以作为新项目的模板起点。

**复利机制**: 每跑完一个 feature,把 "AI 在这个项目里反复犯的错" 加到 CLAUDE.md。这是这套工作流最大的复利点。

### 2. Figma MCP 已接入
确认 Claude Code 配置里有 Figma MCP (你当前环境已启用,deferred tools 里有 `use_figma` 等)。这让 AI 能直接读节点、变量 (颜色/字体/间距)、截图,避免手动贴图丢失设计 token。

### 3. Design Token 中间层 (强制 Figma 一致性)

项目里建立 Theme 常量层 (例: `Voca/Modules/Theme/Spacing.swift`、`Theme/Color.swift`、`Theme/Font.swift`),把 Figma 变量映射成 Swift 常量。

CLAUDE.md 必须有这条硬约束:
> **UI 间距 / 颜色 / 字体必须用 Theme 常量,禁止 magic number (0 除外)。新增间距值前先到 Theme 里建常量,常量值来自 Stage 1 plan 的 token 清单。**

效果:
- AI 不能写 `padding: 13` (因为不在 Theme 里)
- 必须先建常量,常量值有源头 (Figma 变量名)
- review 时 grep magic number 就能发现违规

### 4. PRD 模板 (你已有,持续优化)
保持现有固定模板。**复利机制**: 每次 AI 在 plan 阶段问出来的问题,如果是 "PRD 没说清",就把这个字段补进模板。

测试同事的"边界用例清单"应作为 PRD 的固定一节 — 不是事后让 QA 补,而是 Stage 0 就一起准备。

## 工作流: 一个 feature 的完整流程

### Stage 0: 输入准备 (人, 离线)

产出物:
- PRD PDF (按模板填完)
- Figma 链接 (一个,包含整个 feature 所有页面)
- 关键约束清单 (deadline / 必须复用的现有组件 / 性能要求)

### Stage 1: Plan 会话 (AI 规划 + 人审阅)

**启动**:
```
cd <project>
claude
> Shift+Tab 切到 plan mode
> "PRD: <粘贴或附件>。Figma: <链接>。这是一个新 feature,
   先读 CLAUDE.md 和相关现有代码,然后给我 plan。卡住就问。"
```

**AI 在 Stage 1 做的事 (5 步,缺一不可)**:

**1) 输入吸收**
- 读 PRD + 通过 Figma MCP 读 Figma + 读 CLAUDE.md
- 通过 Figma MCP `get_variable_defs` 提取**所有** spacing / color / font token (不允许目测截图)

**2) 代码考古 → 出文字报告 (不直接做决策)**
- 涉及哪些 ViewController / View / Model / Service?(贴文件路径)
- 当前数据流向是什么?(从 API 到 UI 完整链路)
- 哪些现有组件可复用?

**3) 技术选型清单 (人看报告决策,不让 AI 拍板)**

复杂需求 (前后端联动) **每一项都要写明**:
- **入口**: 前端从哪个 view / 哪个交互触发?
- **前端调用链**: 哪个 ViewController → 哪个 Method → 哪个 Service → 哪个 API
- **数据模型变更**: 哪些 Model 增删改字段?是否影响缓存?是否破坏向前/向后兼容?
- **接口变更**: 改哪几个接口?字段?请求/响应结构?字段命名约定 (snake_case vs camelCase) 是否与现有一致?
- **改动后数据流向**: 接口下发字段 → 网络层 → 模型 → 缓存 → ViewModel → View 完整链路
- **状态管理**: 涉及哪些 state?**必须画状态机图** (Mermaid),标清每个 transition 的触发条件
- **异常处理**: 失败 / 超时 / 网络断开 / 缓存失效 / 服务端错误码 各自如何表现?
- **副作用**: 改动会影响哪些其他模块?(grep 验证,贴 grep 结果)
- **依赖项**: 后端是否同步排期?第三方 SDK 是否需要升级?
- **回滚预案**: 上线后出问题怎么回滚?要不要走灰度?

**4) 边界用例预演**
- 列举所有用户旅程 + 每条旅程的异常分支
- 列举所有 state 转换条件,标注 "已覆盖" 和 "未明确(需提问)"
- 边界用例最好引用 QA 在 Stage 0 提供的清单

**5) 产出 plan,必须包含**:
- Design token 清单 (Figma 变量 → 项目 Theme 常量映射)
- 文件改动清单 (新增 / 修改 / 删除,贴路径)
- 子任务拆分 (Stage 2 一个会话对应一个子任务)
- 复用决策 (引用具体文件路径)
- 测试用例清单 (单元 + API 配置 + 关键状态机分支)
- 状态机图 / 序列图 (Mermaid)
- 假设项 / 风险点 / 待澄清项

**人在 Stage 1 做的事**:
- 审 plan,关注: 架构合理性 / 复用是否充分 / 拆分粒度是否合适 / 假设项是否成立
- **重点审异常分支和状态机**: 网络偶发失败 / 业务真失败 / 轮询超时 / 缓存失效 各自如何流转?这是 AI 最容易笼统处理的地方
- **不点橡皮图章**;打回时说清楚 "哪里不对、为什么不对"
- **找资深同事 30 分钟过一遍 plan** (尤其是你不熟悉的模块或第一次做的需求类型 — 如轮询、长任务、推送)。这是借经验最有效的方式,事前借比事后救火便宜得多。也可以参考公司其他 app 同类实现
- 通过后,把 plan 保存到项目内 `docs/plans/<feature-name>.md` (checked-in)

### Stage 2: 实现会话 (AI 实现,每个页面/模块一个独立会话)

**为什么拆多会话**: 避免 context 漂移和压缩丢失。每个会话职责单一。

**启动 (每个子任务一次)**:
```
cd <project>
claude  (新会话,不开 plan mode)
> "执行 docs/plans/<feature>.md 里的子任务 #N (例如:实现 ChallengeListPage)。
   读 CLAUDE.md。卡住就问。"
```

**AI 在 Stage 2 做的事**:
1. 实现该子任务
2. 写单元测试 + API 配置测试 (按 plan 中的测试清单)
3. 自检: 编译通过 + 测试通过
4. 卡住时**提问**,不猜
5. 产出: 改动列表 + 测试用例文件

**人在 Stage 2 做的事**:
- **审测试用例** (是否覆盖关键路径、状态机分支、边界值)
- 不逐行审代码 (low-value, 编译和测试已经验证)

### Stage 3: 真机/模拟器验收 (人)

**前置: 后端先跑通再联调**
- 涉及后端改动时,先把后端代码在本地跑通,亲自看一眼 "后端到底返回了什么 JSON" — **不要用 postman 替代真机测试**
- postman 只能验后端单点,**真机才能暴露端到端问题** (序列化、字段命名、网络环境、缓存)
- 找测试同事拿完整可复现操作流程,而不是自己想当然走 happy path

**正式验收**
人在 Xcode 里跑起来,验收 UI 和交互。发现问题:
- UI 偏差 → 回到对应 Stage 2 会话,具体描述偏差,要 AI 修
- 行为错误 → 同上,但优先要求 AI 写复现该错误的测试,再修

接口返回内容验证: 你先在真机/模拟器跑一遍,把 charles / 控制台日志 / 后端日志贴给 AI,AI 帮你判断是否符合预期。**AI 不能替你跑真机**。

## MCP / Skills 速查 (你当前可用的)

按 Stage 选最相关的几个,不一一列举:

| Stage | 推荐 Skill / MCP |
|---|---|
| Stage 0 (输入准备) | `pdf` (读 PRD)、`product-management:write-spec` (PRD 模板设计) |
| Stage 1 (Plan) | `figma-implement-design` 或 `figma` (Figma → 代码翻译)、`engineering:architecture` (重大技术选型 ADR)、`engineering:testing-strategy` (定测试策略)、`context7` (查第三方库官方文档),按领域选 `axiom-*` (做 IAP → `axiom-in-app-purchases`、做网络 → `axiom-ios-networking`、做 SwiftUI 架构 → `axiom-swiftui-architecture`) |
| Stage 2 (实现) | 领域 `axiom-*`、`xcode-build-fixer` (构建错)、`swift-testing` (Swift Testing) |
| Stage 3 (验收 / 出 bug) | `engineering:code-review` (自审)、`engineering:debug` (结构化排查) |

**调用方式**: 输入 `/<skill-name>` 或者直接描述需求,AI 会按上下文自动调用。

**怎么决定要不要用**: 不强求每个 Stage 都用。判断标准是 — 任务是否进入了 skill 的描述范围。例如做 IAP 时 skill 里有踩坑经验,纯 vanilla SwiftUI 列表就不需要查。

## 失败兜底

| 情形 | 处理 |
|---|---|
| PRD 歧义 / Figma 缺失 / 技术选型不明 | AI 提问,人在对话里回答 |
| AI 重复犯同一错误 ≥ 2 次 | 人介入,具体指出问题或直接手动修;事后把这个 case 加到 CLAUDE.md |
| Plan 在实现阶段需要重大调整 | 回到 Stage 1 重新 plan,不在 Stage 2 临时改架构 |
| 测试持续失败 | 打回 AI,要求解决根因,**禁止** `--no-verify` 或绕过测试 |

## Bug 复盘模板 (针对 "症状相同 + 根因不同" 的连环 bug)

复盘清单:

1. **画完整序列图**: 前端 → 后端 → 第三方 → 后端 → 推送 → 前端轮询 / 接收。每个节点列 "可能失败的方式"。这一步在 Stage 1 就该做,补做就是补债
2. **找 QA 拿完整可复现操作流程**: 不要只信你自己的复现路径,也不要直接判断为"偶现 bug" 放过 — "偶现" 通常意味着复现条件 (如网络抖动) 还没找全
3. **排查路径要顺藤摸瓜**: 从前端入口开始一路查到后端,确认每一步传递的参数是什么,而不是凭直觉跳到某个仓库
4. **问最深的问题**: "症状 = 根因 A?" 还是 "症状 = 根因 A or B or C?" — 修第一次时就让 AI 把**所有可能导致这个症状的根因**列出来,而不是单点修
5. **状态机优先**: 大多数 "症状相同根因不同" 的连环 bug,根源都是 plan 阶段状态机不完整。回到 plan 阶段把状态机重画,而不是在代码里打补丁
6. **每修一次必加一个测试 case 防回归**

### 关键洞察: AI 协作下的盲区类型

| 盲区 | 例子 | 怎么补 |
|---|---|---|
| **指令笼统盲区** | 你说 "接口失败 = UI 失败",AI 完美执行,但状态机里 `网络偶发失败` 不该等于业务失败 | Stage 1 强制画状态机,plan 模板加 "网络异常处理策略" 一栏 |
| **经验盲区** | 你不知道 "轮询接口失败应该重试不是失败" 这种行业惯例 | Stage 1 找资深同事 30min review;同类需求先看公司其他 app 怎么做 |
| **协作盲区** | 后端没写接口文档你自己接,字段拼写没 check | 这是个人习惯问题,养成排查时顺藤摸瓜的习惯,先怀疑前端再怀疑后端 |
| **联调盲区** | 用 postman 替代真机测试,后端模型名错没暴露 | Stage 3 强制 "本地跑通后端 + 真机测",postman 只能用于辅助 |

复盘产出物 (写到 docs/postmortem/ 里):
- 状态机最新版
- 补的测试 case
- CLAUDE.md 增补条目 (例如 "异步任务模式下,query 轮询失败默认重试,只有业务字段返回 failed 才进入失败 state")

每个 feature 交付前必须全部满足:
- ✅ 编译通过 (`xcodebuild` Debug 无错误)

## 验证 (怎么测试这套工作流是否真的提效)

1. **基线**: 选一个 Voca 最近完成的 feature (例如近期的 paywall 改造) 作为对照
2. **试跑**: 用本工作流跑一个等量级新 feature (Voca 现有项目即可)
3. **测量**:
   - 你介入手动改动的次数 (越少越好,但不是 0)
   - 是否出现绕过项目约定 (grep CLAUDE.md 里的硬约束,看有没有被违反)
   - PR 大小是否合理 (太大说明 plan 拆分不够)
   - 主观感受: 是否比纯手写或纯 vibe-code 都更省力
4. **迭代**:
   - 把 AI 反复犯的错 → 加到 CLAUDE.md
   - 把 PRD 经常缺的字段 → 加到 PRD 模板
   - 把 plan 阶段总要追问的事 → 加到 PRD 模板或 Stage 0 准备清单

## 不在本工作流范围内 (有意省略)

- **Hook 自动 build/test**: 可以后续加,但前期保持简单,人工跑
- **Skill 化封装** (例如 "PRD → plan" 一键 skill): 等工作流本身稳定后再封装
- **多 agent 并行执行**: 等单流程跑顺再优化
- **Linear / Jira 等任务系统集成**: 不在第一版

## 一句话总结

**人提供清晰的 WHAT (PRD + Figma + 约束),AI 负责 HOW 的规划和实现,在 plan 和测试用例两个节点强制人工 review,真机验收人工兜底。复利点是持续打磨 CLAUDE.md 和 PRD 模板。**
